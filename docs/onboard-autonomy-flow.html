Onboard Autonomy Flow

The onboard autonomy stack treats the car as a black box: capture sensor input, convert it into evidence, retain useful memory, decide whether to act, and send a bounded vehicle action. In the current memory-first rewrite, the decision model can emit a null action, so the car remains idle while perception and memory are validated.

Source points: autonomy/vehicle/, autonomy/perception/, autonomy/decision/, autonomy/runtime/, implementations/operations/

Main Runtime Loop

Vehicle Boundary

The onboard code talks to the car through CarInterface, not directly to autonomy internals.

  • read_sensors() returns a sensor snapshot.
  • Current sensor: front_camera.
  • execute_pulse() sends bounded motion.
  • stop() is always available.

Perception Evidence

Perception turns image data into evidence, not final world truth.

  • core: brightness, contrast, image size.
  • traversability: floor / occupied evidence.
  • motion: relative feature groups when frame pairs exist.

Decision Memory

Memory stores frames, keyframes, events, and perception evidence with limits attached.

  • FrameMemory keeps the current observation.
  • KeyframeMemory keeps important frames.
  • Events record commands, failures, and confidence changes.

Decision Model

The next layer reads memory and emits either no action or a vehicle action proposal.

  • Current rewrite target: null action.
  • Future: motive, proposal, confidence, action path.
  • Simulator truth is not part of this input.

Action Commit

The action output is adapted back to the black-box vehicle boundary.

  • null: hold idle / safe stop.
  • VehiclePulse: forward, reverse, steering, duration.
  • Command result feeds back into memory.

On PiRacer

DonkeyPiCar uses the Donkey server on 127.0.0.1:8887 when running onboard, or piracer.local:8887 from a local machine.

In Simulation

ChaseSimCar now exposes the same black-box surface: capture front view, execute action, stop. Extra world state stays out of autonomy input.

Artifacts

Runs write inspectable files under lab/runs/: frames, memory JSON, summaries, overlays, and debug images.

Current Memory-First Behavior

Capture

The car or simulator provides a SensorSnapshot. Right now that snapshot contains one current front-view frame.

Perceive

The image is converted to a single_frame_memory_observation_v0 record. With one frame, relative motion remains null.

Store

The observation is written into AutonomyRunMemory as one FrameMemory and one keyframe.

Choose Null Action

The first decision-model rewrite can return no action while memory quality is inspected.

Remain Idle

A null action maps to hold/stop. This keeps onboard tests safe while the memory layer is built out.

Important boundary: perception can say “there is occupied traversability evidence,” but memory should not call that an object until later frames, motion evidence, or a stronger detector supports that interpretation.

Layer Contracts

Layer Input Output Current Code
Vehicle Commands from operations or decision output. Captured frame files, command metadata, safe stop. autonomy/vehicle/vehicle.py
Perception Images or image pairs. Frame facts, traversability evidence, feature/motion evidence. autonomy/perception/core, features, motion, traversability
Memory Perception observations plus command/event history. Frames, keyframes, retained evidence, event trace. autonomy/decision/memory.py
Decision Memory snapshot and current observation. Idle control now; later concrete engines can select movement. autonomy/decision/cycle.py, autonomy/runtime/engine.py
Operations Vehicle plus plan or procedure. Reusable run artifacts and reports. implementations/operations/
