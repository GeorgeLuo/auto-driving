Milestone 004: Physical Perception Parity

Prove that the PiRacer continuously interprets its own camera while movement authority remains manual, and make each latest structured result visually inspectable against the exact frame that produced it. Simulator compatibility is the reference shape, not a substitute for physical evidence.

Status: closed 2026-07-18 Started: 2026-07-16 Record: frozen at closeout Drive mode: manual Action output: zero Persistence: opt-in only Delivery: rolling PR horizon

High-Level Objective

Observe Without Driving

Run physical perception independently from Donkey pilot mode. Manual control remains authoritative and the autonomy engine continues to emit zero steering and throttle.

Pair Evidence With Its Frame

Publish one bounded latest snapshot containing the exact processed frame identity, structured PerceptionText, timing, health, and explicit stale or unavailable state.

Inspect Through Automa

Give the physical vehicle the same concise terminal stream and local frame-overlay view used to understand Chase perception, without putting frontend ownership on the Pi.

Success is a stationary PiRacer pointed at a floor-standing object. An operator can see the current camera frame, a corresponding generic floor boundary at the correct left/center/right location, and zero movement. Success is not semantic recognition or navigation.

Completion Usage

These commands are the proposed public interface and are not all implemented yet. Exact options may sharpen, but these five workflows and their success signals are stable milestone scope.

Deploy And Verify Physical Perception

Starting state: the prepared PiRacer is powered and joining the local network; no separate server-start command is required.

Proposed execution:

./cli/automa vehicles update core --id piracer
./cli/automa vehicles active
./cli/automa vehicles update perception \
  --id piracer --algorithm lightweight_observer
./cli/automa vehicles update autonomy \
  --id piracer --restart
./cli/automa vehicles info perception --id piracer

Success signal: core update confirms the boot-enabled service and HTTP readiness, then output names the deployed release and mapper, reports live frame progress rather than only activation, and confirms manual drive mode.

Stream Current Physical Findings

Starting state: physical perception is deployed and the Donkey server is producing camera frames.

Proposed execution:

./cli/automa vehicles stream perception --id piracer

Success signal: one replacing terminal view reports frame ID, result age, algorithm health, signals, and structured things. It visibly advances while drive mode remains manual. --json returns the same latest snapshot for automation.

Inspect Frame-Matched Overlays

Starting state: the physical stream has published at least one healthy result.

Proposed execution:

./cli/automa vehicles info perception --id piracer
# Open the reported local Perception view URL.

Success signal: the browser shows the exact processed frame, overlay regions and labels from that frame, source/result IDs, lag, age, and independent layer toggles.

Run The Physical Object Check

Starting state: the car is stationary with clear floor ahead and a contrasting floor-standing object available.

Proposed execution:

./cli/automa vehicles perception check \
  --id piracer --record

Success signal: the command prompts for clear, left, center, right, and removed placements, waits for fresh onboard results, and builds a review pairing each finding with its frame. Boundary location follows placement, removal clears current evidence, and every captured control remains zero.

Apply Perception To Saved Frames

Starting state: one image or capture directory is available locally; no vehicle or simulator needs to be online.

Available execution:

./cli/automa vehicles perception apply path/to/images \
  --candidate floor_continuity --record

Success signal: every source image passes through the same perception contract, the report identifies source.kind=apply and the effective candidate configuration, and --record produces a frame-matched review without commanding a vehicle.

CLI usage · candidate guide · 50-frame physical evidence

Scope Boundaries

In Scope

  • Always-on bounded perception while physical drive mode remains manual.
  • Supervised boot-to-readiness behavior for the physical Donkey runtime.
  • Exact latest frame/result publication from the Pi.
  • Physical terminal stream, JSON inspection, and local visual overlay through Automa.
  • Generic floor and boundary evidence from packaged stable plugins.
  • One bounded stateless lab candidate, compared with the packaged baseline on identical physical frames and promoted only through explicit gates.
  • Repeatable stationary physical acceptance and Pi resource measurements.
  • Explicit unavailable, stale, and processing failure behavior.

Out Of Scope

  • Semantic object names, identity, classification, or VLM use.
  • Broad model comparison, FastSAM, unresolved learned-model deployment, or more than one new milestone candidate.
  • Temporal stabilization, persistent tracking, or camera calibration as prerequisites for candidate qualification.
  • Decision memory, pattern inference, projections, or world models.
  • Autonomous mode, steering, throttle, navigation, or collision avoidance.
  • Metric distance, calibration, SLAM, or floor-plan construction.
  • Default recording, remote frontend hosting, or historical perception storage.

Exit Criteria

  1. After one core deployment, Pi boot automatically starts automa-donkey.service, supervises process exit, and reaches read-only /autonomy/status readiness in manual user mode without another command.
  2. The deployed Pi runs the selected perception mapper at a bounded cadence while Donkey drive mode remains user.
  3. Perception execution and failure cannot change movement authority; physical validation reports zero autonomy steering and throttle throughout.
  4. The Pi publishes one bounded latest snapshot containing the exact processed frame, complete detached PerceptionText, frame ID, timestamps, algorithm, timing, and health.
  5. Publication distinguishes absent, warming, healthy, stale, unavailable, and error state without silently reusing old evidence.
  6. vehicles info perception --id piracer reports live physical state and a usable local view URL.
  7. vehicles stream perception --id piracer presents replacing human output and complete machine JSON from the onboard result.
  8. The browser view overlays only frame-matched image-space findings and exposes frame lag, result age, labels, regions, and layer toggles.
  9. The bounded physical check demonstrates clear, left, center, right, removed, and unavailable states with findings that match the declared floor-boundary contract.
  10. floor-plane-v0 is measured as the common-frame control, and floor-continuity-v1 is either promoted after material measured improvement or retained/rejected in lab/ with inspectable evidence; no candidate requires a stable contract change.
  11. A 60-second Pi run publishes at least two fresh results per second with p95 result age no greater than one second, bounded RSS, and measured latency/CPU evidence.
  12. Default perception operation writes no frames, findings, or project-local logs; explicit recording produces one bounded inspectable run, while service diagnostics remain in the system-managed journal.
  13. Chase retains the same stable perception output and presentation concepts without receiving physical-only assumptions.
  14. Closeout records residual physical limitations and activates milestone 005 only after the live Pi evidence is accepted.
