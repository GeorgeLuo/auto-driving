Milestone 004: Evidence Memory Foundation

Give the decision cycle bounded, inspectable continuity across frames. Memory should retain attributed observation evidence without claiming a complete world model, changing perception semantics, or granting movement authority before its lifecycle and value are understood.

Status: active - planning Started: 2026-07-15 Action policy: idle Persistence: memory only by default Hosts: Chase and DonkeyCar Delivery: rolling PR horizon

High-Level Objective

Retain Evidence, Not Truth

Carry selected observations across frames with source, time, confidence, and uncertainty intact. Memory records what was observed and retained, not what definitively exists.

One Lifecycle Across Hosts

Run the same memory contract and implementation selection in the local Chase controller and onboard Pi host without simulator map input or vehicle-specific memory types.

Bounded And Inspectable

Make capacity, expiry, reset, provenance, and current state visible to operators. Default execution remains in memory and writes no frame or history artifacts.

Success is not a detailed floor plan. Success is a stable, economical answer to "what recent evidence is the controller retaining, why, and for how long?" that behaves the same in replay, simulation, and the physical runtime while the car remains idle.

Stage Meaning

SensorSnapshot
  -> PerceptionText       current plugin evidence
  -> Observation          decision-facing representation of one moment
  -> MemorySnapshot       bounded evidence retained across moments
  -> Patterns             recurring relationships inferred from memory
  -> Projections          hypotheses about possible future state
  -> Action               immediate control output (idle throughout this milestone)

Observation

Answers what the current cycle made available. It is frame-scoped, detached for inspection, and may explicitly report missing or degraded perception.

Memory

Answers what attributed evidence remains relevant after one cycle. It owns retention and expiry, not semantic classification or future inference.

Patterns

Answer what relationships recur across retained evidence. They are later derived claims and are not required to make memory useful.

Projections And Action

Answer what may happen and what should be done now. They remain no-op so memory can be evaluated without movement masking its failures.

Memory Contract Constraints

ConcernRequired DirectionNot Yet Assumed
ProvenanceEvery retained claim cites source observation, plugin/evidence identity, coordinate frame, observed time, and update time.That two similar records are the same physical object.
ConfidenceRetain source confidence and make any aggregate confidence rule attributable.That repetition alone turns uncertain evidence into truth.
BoundsCapacity, age, and eviction behavior are finite, configured, visible, and testable.Unbounded history, raw frame archives, or an implicit database.
LifecycleConstruction, update, read, reset, activation replacement, and failure states have one owner.Persistence across process or device restart.
CoordinatesPreserve declared image or relative frames and reject incompatible fusion unless an implementation explicitly transforms them.Calibrated metric geometry or a shared global map.
InspectionReturn stable summaries and structured records detached from live mutable state.Dumping every internal index or implementation cache to operators.
FailureA memory failure is explicit, produces no new retained claim, and cannot produce movement.Silently falling back to stale state as if it were current.

Complexity belongs in the stable lifecycle and safety boundary when it makes implementations smaller. A memory plugin should primarily express what to retain, update, merge, or expire; it should not duplicate activation parsing, status rendering, persistence policy, or host wiring.

Evaluation Contract

Deterministic Sequences

Use synthetic observations to prove recurrence, dropout, conflict, expiry, capacity, reset, failure isolation, and repeatable replay without CV variability.

Simulator Evidence

Run camera-derived observations in controlled scenarios. Hidden scene state may score an implementation but never becomes its memory input.

Physical Evidence

Confirm the same activation, bounded state, inspection, and reset behavior on the Pi while the idle engine retains zero steering and throttle.

Work Plan

1. Settle Stable Memory Values And LifecyclePending
  • Replace public Any memory output with the smallest implementation-neutral value contract.
  • Define provenance, timestamps, bounds, reset, detached serialization, and explicit unavailable/error state.
  • Protect the distinction between current observation, retained evidence, inferred pattern, and projection.
  • Use focused autonomy tests before creating concrete reducers or broad CLI behavior.
2. Define Plugin Execution And ActivationPending
  • Give implementations a narrow update/reset/query shape while the framework owns lifecycle, status, timing, and failure isolation.
  • Use one current activation document and remove superseded paths rather than supporting compatibility aliases.
  • Keep source observations read-only and prevent hidden unbounded state.
3. Wire Equivalent Runtime HostsPending
  • Install the selected memory stage in both the local automation worker and Donkey cycle host.
  • Expose memory to runtime inspection and future engines through one shared snapshot shape.
  • Retain the idle engine and prove memory failure cannot alter control output.
4. Add One Functional Memory ImplementationPending
  • Begin with a simple bounded evidence ledger or recency reducer whose claims can be tested directly.
  • Keep implementation files under the matching implementations/memory hierarchy and make deployment selection explicit.
  • Do not introduce semantic object identity, a floor-plan schema, or generalized fusion until evidence demands it.
5. Build Operator And Evaluation WorkflowsPending
  • Support concise info, status, latest-state streaming, reset, deterministic replay, and opt-in recording through Automa.
  • Exercise controlled observation sequences and at least one camera-derived simulator scenario without map input.
  • Verify the same activation and bounded state on the Pi without commanding movement.
6. Consolidate Findings And ClosePending
  • Record which retained representation proved useful, its cost, and where it failed.
  • Decide whether patterns, projections, or information-gathering action have enough evidence for the next milestone.
  • Preserve unresolved algorithm experiments as bounded synthesis or lab work rather than stable memory contracts.

Scope Boundaries

In Scope

  • Stable memory values, lifecycle, activation, reset, and failure behavior.
  • Simple loadable memory implementations with bounded state.
  • Equivalent local simulator and onboard Pi host integration.
  • Operator inspection, streaming, replay, and opt-in recording.
  • Deterministic sequence tests and controlled simulator evaluation.
  • Non-moving physical activation and state inspection.

Out Of Scope

  • Non-idle decision or action policy.
  • Path planning, navigation, collision avoidance, or map-based control.
  • Metric SLAM, calibrated global coordinates, or a complete floor plan.
  • Semantic object recognition or universal object identity.
  • Durable database, cloud memory, or default frame archives.
  • Model training, VLM integration, or privileged simulator state as input.

Exit Criteria

  1. Stable memory input and output types replace public Any at the decision-cycle boundary without encoding a concrete mapping algorithm.
  2. Observation, memory, patterns, projections, and action have distinct documented meanings and stage ownership.
  3. Memory state has finite capacity and age, deterministic serialization, explicit reset, attributable provenance, and isolated failure behavior.
  4. At least one simple implementation retains and expires structured evidence without claiming semantic or metric world truth.
  5. The same activation and lifecycle run through local Chase and onboard Donkey hosts; neither receives privileged simulator map state.
  6. Automa can stage, inspect, run, stream latest memory, replay a sequence, and reset state with concise human output and complete machine output.
  7. Default execution writes no logs, frames, or memory history; recording is explicit and bounded.
  8. Deterministic tests cover recurrence, dropout, conflict, expiry, capacity, reset, failure, and replay, including long-sequence boundedness.
  9. Controlled simulator evidence and one non-moving Pi check verify equivalent stage behavior while the idle engine emits zero movement.
  10. A closeout states what memory representation proved useful, what remains unverified, and whether later pattern or action work is justified.
