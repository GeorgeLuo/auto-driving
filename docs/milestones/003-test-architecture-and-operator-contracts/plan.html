Milestone 003: Test Architecture and Operator Contracts

Make behavioral change visible before autonomy logic expands. The test suite must mirror code ownership, directly protect settled framework contracts, and verify that Automa gives human operators concise, actionable information without hiding failures or leaking internal noise.

Status: active - implementation Started: 2026-07-13 Production behavior: preserve by default Default suite: deterministic and offline Live validation: explicit opt-in Delivery: rolling PR horizon

High-Level Objective

Tests Follow Ownership

Organize tests by stable autonomy contracts, concrete implementations, CLI behavior, local integration, and live systems so failures point to the responsible layer.

Contracts Are Executable

Protect settled decision, perception, runtime, and vehicle interfaces directly rather than relying on broad CLI scenarios to exercise them incidentally.

Operator Output Has A Contract

Verify that human output prioritizes state, outcome, and recovery while machine output remains schema-valid, complete, and free of prose.

This milestone is not a pursuit of maximum test count or a single line coverage percentage. Success means important behavior has an obvious owning test, regressions fail close to their source, and normal operator workflows provide high-signal output in success, degraded, and failure states.

Test Layers

Unit And Contract

Fast, deterministic tests for one owning module or interface. No subprocesses, network, browser, simulator, or physical vehicle.

Local Integration

Disposable filesystems, staged bundles, isolated plugin workers, fake vehicles, and fake external processes. Included in the default suite.

CLI Scenarios

Black-box command execution against controlled state. Assert exit status and operator-visible meaning rather than internal implementation calls.

Live Systems

Bounded simulator or Pi checks. Never run implicitly, never move physical hardware without a specifically named operation, and report unavailable dependencies clearly.

python3 tests/run.py
python3 tests/run.py --live-sim

The flagless command runs all deterministic unit, CLI, and local integration tests. Topic-specific development runs use standard unittest module selection rather than adding many runner flags.

Operator Output Contract

Output ModeRequired SignalNoise To Reject
Human success Outcome first, affected vehicle or simulator, resulting state, and a next command only when useful. Raw JSON, tracebacks, duplicate summaries, implementation-only paths, hashes, or PIDs without operational relevance.
Human degraded state What is available, what is unavailable, whether the operation can continue, and the most relevant recovery action. Silent omission, generic “none found,” raw socket errors, or a success exit code when the requested outcome was not achieved.
Human failure Nonzero exit, concise cause, relevant state, and recovery guidance when the user can act. Tracebacks and internal exception text unless an explicit diagnostic mode requests them.
JSON One parseable document with a schema, explicit status, and enough structured context for automation. Human headings, progress prose, warnings before the JSON document, or fields that disagree with human rendering.
Long operation Bounded progress at meaningful state transitions and one final result. Silent waiting, per-frame chatter by default, or repeated unchanged status.

Human and JSON output should render the same structured operation result. Tests should expose drift between the two forms; production refactoring is permitted when needed to remove duplicated state calculation.

Scope Boundaries

In Scope

  • Test organization and reusable test support.
  • Direct stable-contract coverage.
  • Human and JSON output behavior.
  • Deterministic CI and coverage visibility.
  • Explicit simulator and Pi live-test boundaries.

Out Of Scope

  • New autonomy behavior or movement policy.
  • Perception algorithm accuracy claims without labeled truth.
  • Exact FastSAM masks or fragile pixel-perfect CV outputs.
  • Mandatory live simulator or hardware tests.
  • Changing test frameworks solely for style.
  • Keeping duplicate runners for compatibility.

Risks And Controls

RiskControl
A test-only refactor changes behavior accidentally.Move and split first, run the complete baseline after each topic, then add new assertions in a separate phase.
Broad fixtures drift away from runtime schemas.Use small builders owned by the contract they represent and assert their serialized shape directly.
Exact text assertions make harmless wording changes expensive.Snapshot stable help surfaces; use semantic assertions for dynamic status and failure output.
Coverage targets reward incidental execution.Prioritize named invariants and mutation-sensitive failure cases; use coverage to find omissions, not as the sole quality score.
Live tests become flaky or move hardware unexpectedly.Keep them bounded, opt-in, explicit about side effects, and separate from default CI.
Test support becomes another application framework.Keep helpers limited to process execution, fixtures, fakes, and reusable assertions; business behavior remains in production code.

Exit Criteria

  1. The owned test tree mirrors autonomy, implementations, and cli, with cross-layer and live tests clearly separated.
  2. No test file combines unrelated command families, framework contracts, and concrete implementation behavior.
  3. Decision cycle, runtime manager, vehicle contracts, perception evidence, plugin execution, and activation have direct invariant tests.
  4. Common human success, degraded, and failure states are tested for useful signal and absence of internal noise.
  5. Human and JSON forms are verified against the same semantic operation state.
  6. The canonical flagless runner executes all deterministic tests without live dependencies or default artifact recording.
  7. Pull requests run the deterministic suite automatically and expose owned-code coverage.
  8. Simulator and Pi checks are bounded, named, opt-in, and document side effects and cleanup.
  9. The old runner and redundant fixtures are removed rather than retained as compatibility paths.
  10. A concise closeout records defects found, contract decisions made, test timing, coverage baseline, and remaining live-system risk.
