Milestone 002: Perception Hardening

Establish a credible perception layer before designing decision memory. The milestone must make simulator and physical-camera experiments feel like the same operation, exercise multiple functional image-to-evidence approaches, and determine which per-frame and temporal representations are reliable enough to carry forward.

Status: complete Primary action: observe Movement authority: unchanged and idle Sources: Chase, PiRacer, recorded sequences Closeout output: recommendation for memory inputs

High-Level Objective

Source Parity

Run the same perception experiment from the CLI against Chase or PiRacer by changing the vehicle id, not the perception interface.

Functional Representations

Turn images into inspectable evidence such as regions, surfaces, lines, motion groups, and persistent scene tracks without requiring semantic object labels.

Evidence For Memory

Close with measured conclusions about which outputs are stable, useful, and affordable enough to become inputs to a later memory milestone.

“Useful” must be tested against a small question set: what evidence is visible now, where is it relative to the vehicle, what area appears traversable, what changed, and what persisted across views? This gives perception a consumer without prematurely designing decision memory.

Target Operator Experience

The required local-processing path differs only by vehicle id:

./cli/automa vehicles perception run
./cli/automa vehicles perception run --id piracer
./cli/automa vehicles perception replay path/to/recorded-run
./cli/automa vehicles perception candidates
./cli/automa vehicles perception setup fastsam
./cli/automa vehicles perception compare path/to/recorded-run

The flagless command observes five frames, prefers a usable simulator when multiple vehicles are active, takes no movement control, and does not retain captures unless --record is supplied. Processing physical frames locally is the parity gate. Before closeout, one lightweight plugin must also run through the onboard Donkey runtime.

Scope Boundaries

In Scope

  • A provider-neutral perception experiment command.
  • Plugin-owned component access with shared file-backed and in-memory camera normalization.
  • Per-frame structure proposals and temporal scene tracks.
  • Recorded-sequence replay and comparable reports.
  • Latency and memory measurements locally and on Pi.

Out Of Scope

  • Decision memory implementation.
  • Autonomous navigation or non-idle action.
  • Metric SLAM, semantic scene graphs, or required VLM access.
  • Perfect object recognition or a finalized world model.

Perception Versus Memory

Temporal plugins may keep bounded working state needed to produce the current perception result. Durable history used by decision policy remains the responsibility of a later memory stage.

Experiment Workbench Contract

Candidate Boundary

New perception candidates are developed as runnable plugins under lab/plugins/perception/<candidate>/. A candidate owns its source, configuration, dependency declaration, model provenance, experiment notes, and any candidate-specific tests; shared worker and contract tests remain in the CLI harness.

Generated State

Weights, virtual environments, downloaded datasets, overlays, timing traces, and run artifacts remain ignored. A run records the plugin revision and configuration needed to reproduce it without committing generated data.

Touch Budget

After a one-time generic lab runner exists, an experiment should normally change only its candidate directory. Stable autonomy interfaces, vehicle implementations, and production plugin registration are not edited to accommodate each experiment.

Promotion

A candidate moves to implementations/perception/ only after its output fits the stable perception contract, its dependencies and license are acceptable, and replay evidence meets declared quality and compute thresholds.

FastSAM Candidate Experiment

Hypothesis

Class-agnostic instance masks can provide more coherent current-frame region proposals than hand-tuned lines or contours, without semantic labels, training, map access, or a VLM.

First Implementation

Use pretrained FastSAM-s with the “everything” path at 640-pixel input. Convert every accepted mask into generic region evidence: normalized bounds, centroid, area fraction, contour, confidence, image zone, and whether it reaches the lower image region.

Meaning Boundary

A mask means “coherent region proposal,” not obstacle, traversable area, object identity, distance, or depth. Those interpretations require separate evidence or later aggregation. Missing, empty, and failed inference remain distinct outcomes.

Temporal Follow-Up

Keep segmentation stateless first. If masks are useful, associate keyframe masks across intermediate frames with mask overlap, sparse flow, and bounded track state rather than coupling model inference to every camera frame.

ConstraintDecisionReason
Model FastSAM-s, not FastSAM-x The maintained integration exposes pretrained automatic masks in a roughly 24 MB model; the smaller variant is the appropriate feasibility probe.
Training None in this milestone The pretrained model is already class-agnostic, and the maintained FastSAM API supports inference and export but not training. Collect evaluation data before considering adaptation.
Runtime Local CPU baseline, optional Apple MPS comparison Local processing is the primary path. The Pi supplies frames until target-hardware measurements justify moving inference onboard.
Cadence Bounded stills, then low-rate keyframes First establish mask usefulness and latency. Continuous inference is not a prerequisite for continuous perception if cheaper tracking bridges keyframes.
Deployment PyTorch reference, then exported runtime benchmark Only attempt ONNX or NCNN after reference outputs are stable; exported output parity and post-processing cost must be measured.
Dependencies Candidate-local optional environment Do not add PyTorch or Ultralytics to core requirements merely to evaluate one plugin.
License Promotion gate The maintained Ultralytics implementation and models carry AGPL or commercial-license considerations. Record exact code and weight provenance before production adoption.

Published timing is directional, not a target-machine result. The maintained documentation reports about 58 ms per image for FastSAM-s on a newer Apple M4 test machine, while Raspberry Pi guidance shows even much smaller nano models slowing materially on ARM. The M1 and Pi 4 must be benchmarked directly before assigning a cadence.

Simulator Control

Physical Scene Sequences

PiRacer captures with boxes, floor, walls, shadows, texture changes, and bounded observer motion. Compare the same plugin configuration and output schema used in simulation.

Final Findings

ApproachObserved ValueCurrent Disposition
Floor boundaries On physical replay, boundaries follow the wall/floor frontier and split around approaching boxes. On recorded Chase, center boundaries move nearer after forward motion, shift with left/right views, and return after reverse. Production lightweight path. Carry as non-semantic boundary evidence; never relabel it as an object without another source.
OpenCV color regions Cheapest coherent-region candidate, strong adjacent-frame continuity, no model or license dependency, but texture and lighting can split one surface. Promotion candidate. Await Pi timing and controlled recall review before moving out of the lab.
FastSAM-s Visually cleaner whole-surface masks in physical and depth-obstacle frames, but roughly 640-673 MiB peak RSS and 141-207 ms warm median on M1, with an Ultralytics license gate. Retain as local diagnostic. Do not deploy onboard or promote into core requirements in this milestone.
Feature-motion tracks Stable run-local tracks capture broad expanding/contracting scene layers, which is useful depth-change evidence. They are not object identities and cost about 1.95 seconds per physical frame locally. Retain as offline diagnostic. Do not include in the lightweight onboard chain.
VLM prep Produces deterministic review inputs but no perception meaning on its own. Diagnostic only. It is unavailable when artifact recording is disabled and should not enter memory.

The likely memory boundary is structured current evidence: source plugin/run status, thing kind, image-relative location, confidence, and bounded properties such as track support or floor-boundary width. Raw prep artifacts, representation-health scores, and run-local track ids should not become durable world facts. Pi measurements confirm that lightweight current evidence is viable at a slower cadence, while heavier temporal and segmentation diagnostics should remain local or asynchronous.

Logical Omissions Addressed

Falsifiability

A representation looking plausible is not enough. Simulator truth may be used only as held-out evaluation, never perception input. Physical runs need controlled scenes or lightweight human annotations so failures can be identified.

Observer Motion

Long-running scene understanding cannot be assessed only from a fixed camera. The corpus needs stationary, forward, reverse, and turning sequences, captured through separate bounded operations rather than decision output.

Experiment Distribution

Success in one room or simulator scene is not hardening. The milestone needs variation in lighting, texture, lens effects, blur, object layout, and camera-to-subject distance.

Selection And Retirement

Candidate plugins need explicit dispositions at closeout: promote, retain as diagnostic, or retire. Fusion should be tested only after each source of evidence is characterized independently.

Unknown Versus Absent

No evidence, unavailable input, low confidence, and analysis failure are logically different outcomes. Memory cannot make sound decisions if perception collapses them into one false value.

Temporal Boundary

Perception may keep bounded working state to describe the current scene. It must not quietly become durable decision history; the milestone closeout should specify exactly what the later memory layer receives.

Viability includes computation cost. An approach that is accurate only with unbounded latency or memory is not a plausible perception input for the vehicle, even if it remains useful as an offline evaluator.
