Automation Engine Initial Commit Backlog

The target initial commit should establish the shape of the automation engine without pretending the driving logic is finished. The loop should be real, the stage interfaces should be stable, movement should remain gated, and concrete behavior should live behind loadable implementations.

Goal: initial repo commit readiness Default action: idle Primary test target: Chase simulator Physical target: PiRacer via DonkeyCar bundle Verification: 48 tests plus physical idle deployment passing Directory contract: JSON

North Star

Vehicle Inputs

Read generic sensor snapshots from a vehicle adapter. Today this is front camera only, but the shape should allow more sensors later.

  • SensorSnapshot
  • timestamp and vehicle metadata
  • last action status when available

Observe

The observe stage calls perception implementations and returns current-cycle evidence. It does not own memory or movement.

  • PerceptionMapper
  • Observation
  • uncertainty and limits

Remember

Memory stores the useful state between cycles. For the first commit this can be latest-only plus small history.

  • latest observation
  • recent signals
  • optional keyframes

Act

The action stage returns idle by default. Movement requires an explicit engine and runtime motion gate.

  • VehicleAction
  • AutonomyControl
  • debug snapshot

The simulator's staged loop is the architectural reference, but its privileged map access is not portable. The repo should copy the stage ordering and debug-cycle shape, not the simulator's direct world-state reads.

Iteration Boundaries

In Scope

  • Generic staged controller cycle.
  • Perception as an implementation-backed observe stage.
  • Idle action as the only default movement behavior.
  • CLI-driven deployment, activation, status, and tests.
  • Directory boundaries that separate core, implementations, deploy, runtime, and lab artifacts.

Out of Scope

  • Full autonomous navigation.
  • Semantic object recognition as a dependency.
  • Continuous SLAM or calibrated metric world modeling.
  • Simulator-source changes for the external controller.
  • Physical movement by default when automation starts.

Design Principle

  • Every stage can no-op.
  • Every implementation can be swapped.
  • Every active runtime is inspectable through CLI.
  • Every generated artifact lives under runtime/ or lab/.
  • Every deployment path is reproducible through ./cli/automa.

  • No compatibility import paths or deprecated CLI aliases.
  • Hashed controller releases with manifests and activation metadata.

Directory Target

Canonical contract: docs/directory-structure.contract.json This interactive tree is the human-readable view of that directory-only contract.
  • auto-driving/workspace root
    Local source of truth for automation engine code, concrete implementations, deployment bundles, generated runs, and CLI access.
    • autonomy/stable kernel
      Stable contracts, stage shapes, reusable primitives, and generic runners. Concrete vehicles, algorithms, and operation recipes belong in implementations/.
      • vehicle/black-box car interface
        Vehicle-agnostic interface for sensor reads, pulses, actions, capabilities, and adapter boundaries.
      • perception/perception contracts
        Stable perception interface and shared primitives. Active algorithms live in implementations/perception/.
        • core/frame primitives
          Frame observations and low-level image comparisons with no scene-specific policy.
        • features/feature primitives
          Reusable feature tracking primitives.
        • motion/motion primitives
          Reusable relative-motion and motion-group primitives.
        • traversability/floor primitives
          Reusable floor and traversability primitives.
        • landmarks/landmark primitives
          Reusable landmark and distance primitives.
        • mappers/mapper entrypoints
          Mapper loaders and stable mapper entry points that assemble configured implementations.
      • decision/staged controller cycle
        Generic observe, memory, pattern, projection, action, and snapshot cycle. Stages should be optional so the engine can run observe-only or idle-only.
      • runtime/engine host
        Generic runtime contracts for loading engines, stepping them, applying mode gates, reporting status, and hosting controller loops.
    • implementations/concrete implementations
      Concrete code that satisfies autonomy/ contracts. This tree mirrors autonomy/ so inspection and promotion paths stay obvious.
      • vehicle/vehicle adapters
        Concrete vehicle adapters implementing autonomy/vehicle contracts.
        • picar/physical vehicle
          PiRacer-specific HTTP, DonkeyCar, and hardware assumptions.
        • chase_sim/sim adapter
          Metrics UI / Chase simulator adapter exposing only vehicle-like I/O to autonomy code.
      • perception/sensor-to-evidence
        Concrete perception mappers and plugin-style sensor-to-evidence algorithms. Current plugins are file-based within this directory.
      • decision/stage implementations
        Concrete non-default decision engines and stage implementations. The stable idle default belongs to autonomy/runtime/.
      • runtime/runtime hosts
        Concrete runtime hosts for environments that need special integration code.
        • donkeycar/Donkey host
          DonkeyCar part and drive-loop integration.
      • operations/bounded workflows
        Concrete operation plans and workflows.
        • startup_action_check/command validation
          Action-and-capture checks to confirm commands register.
    • deploy/remote targets
      Source-controlled deploy bundles for physical or remote runtimes. Target-specific deployment material is nested under targets/, which is different from generated runtime bundles under runtime/.
      • targets/deployable targets
        One subdirectory per remote target family. This keeps future physical or remote integrations parallel to the current DonkeyCar target.
        • donkeycar/PiRacer target
          The physical PiRacer deployment bundle, including app resources, vendor source material, and local patches.
          • app/remote app root
            Files synced to the car's app/runtime directory.
          • vendor/generated checkout
            Generated pinned DonkeyCar checkout recreated from the source-controlled vendor manifest and patches; excluded from commits.
          • patches/target patches
            Local changes applied to vendor or target code during deploy preparation.
    • cli/user access point
      The primary command surface. Common user workflows should be here instead of duplicated scripts.
      • commands/command groups
        Vehicle, simulator, deploy, implementation, runtime, and diagnostic commands.
      • services/command helpers
        Shared command business logic that can be tested without shelling through the CLI entrypoint.
      • tests/CLI harness
        Common scenario tests, with offline defaults and explicit live-simulator coverage.
    • runtime/generated local state
      Generated and mutable local state. This should be disposable and reproducible from source plus CLI commands.
      • vehicles/per-vehicle state
        Local runtime directories keyed by discovered or configured vehicle id.
        • <vehicle-id>/runtime instance
          One local runtime area for a simulator or physical vehicle.
          • bundle/staged controller
            Generated controller bundle with extracted code, versioned releases, active configuration, and worker state.
            • autonomy/extracted core
            • implementations/extracted implementations
            • releases/archives and manifests
            • runtime/activations and worker state
          • deploy/target staging
            Target-specific activation metadata staged for physical deployment.
      • tmp/scratch
        Short-lived temporary output created by CLI or local processing.
    • lab/experiments and runs
      Unstable self-contained experimentation space. Anything here should be allowed to churn without changing the core architecture.
      • experiments/frozen experiments
        Experiment code, configs, notes, and artifacts kept together for review or later extraction.
      • runs/generated runs
        Captured images, reports, overlays, and one-off generated analysis output.
    • frontend/local UI surfaces
      Local UI source that belongs to this project. Simulator UI stays in Metrics UI, so this should remain specific to local vehicle-facing surfaces.
      • donkeycar/PiCar UI
        Frontend files specific to the DonkeyCar/PiRacer control surface.
    • docs/architecture notes
      Human-readable architecture docs, diagrams, backlog summaries, and decision notes.
    • scripts/transitional helpers
      Low-level helpers that are not user workflows. Anything overlapping with ./cli/automa should be removed or folded into CLI services.
      • calibration/calibration tools
        Calibration and measurement scripts not yet promoted into operation implementations.
      • decision/decision tools
        Decision inspection helpers not yet promoted into CLI.
      • perception/perception tools
        Perception inspection helpers not yet promoted into CLI.
      • deploy/remote helpers
        Remote helper scripts called by CLI deploy flows.

The main distinction is stable kernel versus concrete implementation versus generated state. Source code belongs in autonomy/, implementations/, deploy/, frontend/, and cli/. Generated captures, bundles, and reports belong in runtime/ or lab/.
